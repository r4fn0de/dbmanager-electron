# Plan: AI Context for Selected Table in Tables Section

## Goal

When the user is on the **tables section** and selects a table, send that table's context (schema, columns, PKs, FKs) to the AI chat — similar to how the SQL editor sends selected SQL text. Show a context chip in the AI chat indicating which table is selected.

## Architecture

The existing flow is:
1. `SqlEditor` calls `useAiChatGlobalStore.setSqlContext()` with schema + editor state
2. `__root.tsx` reads `currentContext` from the store → passes to `AiChatPanel`
3. `AiChatPanel` shows context chips (`selectionPreview`, `errorPreview`)

The tables section currently **never calls `setSqlContext`**, so the AI chat has no table context when browsing tables.

## Changes

### 1. Extend `AiChatContextPreview` — `src/lib/stores/ai-chat-global.ts`

Add a `tablePreview` field:

```ts
export interface AiChatContextPreview {
  connectionLabel: string;
  dbType: DatabaseType;
  selectionPreview?: string;
  errorPreview?: string;
  tablePreview?: string; // NEW — e.g. "public.users (12 columns, 3 FKs)"
}
```

No store logic changes needed — `setSqlContext` already accepts any `contextPreview` object.

### 2. Send table context from `DatabasePageContent` — `-DatabasePageContent.tsx`

Add a `useEffect` that calls `setSqlContext` when:
- `activeSection === "tables"`
- `selectedTableRef` is non-null
- `selectedTableDetails` is loaded

Build a context string containing:
- Table name + schema
- Column list with types, nullable, PK/FK markers
- Indexes
- Foreign keys

Build a preview string for the chip: `"schema.table_name (N columns)"`

Call `clearSqlContext` when switching away from the tables section or deselecting the table.

This follows the exact same pattern as `SqlEditor` — a `useEffect` keyed on the relevant state, calling `setSqlContext` with a unique `sourceId`.

### 3. Read table context in `__root.tsx`

Pass `storeContext.contextPreview?.tablePreview` to `AiChatPanel`. Add `tablePreview` to the `contextPreview` prop shape — the prop is already an inline type, just add the field.

### 4. Show table context chip in `AiChatPanel.tsx`

Add a third context chip (after selection and error chips) for table context:

- Shows when `contextPreview?.tablePreview` is non-empty and not dismissed
- Icon: `table` icon (matches the tables section)
- Label: "Selected Table"
- Same dismiss behavior and spring animation as existing chips
- Same auto-reset when context changes

Also include the table schema context in the AI system message when building `schemaContext`.

## Implementation details

### DatabasePageContent context builder

```ts
const tablesContextSourceId = `tables-${connectionId}-${...}`;

useEffect(() => {
  if (!isRouteActive || activeSection !== "tables" || !selectedTableRef || !selectedTableDetails) {
    return;
  }

  // Build schema context string from selectedTableDetails
  const ctx = buildTableContextString(selectedTableRef, selectedTableDetails);
  const preview = `${selectedTableRef.schema}.${selectedTableRef.name} (${selectedTableDetails.columns.length} columns)`;

  setSqlContext(tablesContextSourceId, {
    connectionId,
    connectionLabel: ...,
    dbType,
    schemaContext: ctx,
    contextPreview: {
      connectionLabel: ...,
      dbType,
      tablePreview: preview,
    },
  });

  return () => clearSqlContext(tablesContextSourceId);
}, [activeSection, selectedTableRef, selectedTableDetails, ...]);
```

The `schemaContextForAi` already has full schema info, so the table context can focus on just the selected table's details for more targeted AI help.

### Context priority

When the user is on the **SQL editor** section, `SqlEditor` owns the context (editor SQL + selection + errors).
When on the **tables** section, `DatabasePageContent` owns the context (selected table + schema).
The store uses `currentSqlContextOwner` to track which source owns the context — only one source is active at a time, preventing conflicts.

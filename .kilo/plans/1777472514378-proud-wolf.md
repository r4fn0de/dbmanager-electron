# Análise: Migração de partes do backend para Rust

## Veredito

**Migração completa não compensa.** O backend é I/O-bound e os drivers já são nativos em C/C++. Uma migração incremental focada em gargalos específicos (export de dados, SQL parsing) pode fazer sentido se houver problemas reais mensurados.

## Revisão do Relatório Proposto

### Pontos corretos do relatório

- Tabela "onde Rust faz diferença real vs irrelevante" é precisa
- Observação sobre overhead de serialização IPC é crucial
- Estratégia incremental ("começar com TS, migrar gargalos") é a mais sensata
- Reconhecimento de que UI/state/tabs não se beneficiam do Rust

### Problemas identificados

1. **Estrutura proposta ignora o codebase existente** — Propõe `electron/` e `src/` novos, mas o projeto usa `src/features/shell/`, `src/ipc/`, `src/routes/` com TanStack Router file-based routing e oRPC. Migração real precisa mapear para estrutura existente.

2. **`bollard` (Docker via Rust) é overkill** — Projeto já usa `embedded-postgres`. Se precisar de Docker, `dockerode` no Node.js resolve sem adicionar complexidade Rust.

3. **`klickhouse` é instável** — `@clickhouse/client` atual funciona bem via HTTP. Migrar para crate menos maduro introduz risco sem ganho claro.

4. **Cross-compilation em CI não é detalhada** — Distribuir `.node` nativo em macOS (arm64 + x64), Windows (x64 + arm64) e Linux (x64) requer múltiplos CI runners ou `cross`. Muito mais complexo que o build atual.

5. **Testes não são mencionados** — Projeto usa Vitest + Playwright. Testar camada Rust requer `#[cfg(test)]` para unitários, mas testes de integração napi-rs precisam de Node.js rodando — não trivial.

6. **`sqlx` compile-time checked perde utilidade** — Para um DB manager genérico que conecta a qualquer banco, o benefício de type-safety do `sqlx` se perde; a maioria das queries será `sqlx::raw_sql()`. É essencialmente o mesmo que usar `pg`/`mysql2` mas em Rust.

## Razões contra migração completa

1. **I/O-bound, não CPU-bound** — Node.js já é excelente para operações assíncronas de rede. Rust não acelera queries remotas.
2. **Drivers já são nativos** — `pg`, `mysql2`, `better-sqlite3`, `ioredis` são C/C++ bindings. Não há ganho em reescrever o wrapper em Rust.
3. **Complexidade de build** — Cross-compilation via NAPI-RS complica `forge.config.ts` e CI/CD.
4. **Dois ecossistemas** — Bun + Cargo = dois sistemas de dependência, dois toolchains.
5. **Tamanho do pacote** — Binários Rust aumentam significativamente o tamanho do app Electron.

## Onde Rust faria sentido (pontos isolados)

| Área | Problema atual | Ganho esperado |
|------|----------------|----------------|
| SQL parsing para autocomplete | `node-sql-parser` é lento para queries complexas | `sqlparser-rs` 10-100x mais rápido |
| Export de resultsets grandes (500k+ rows) | JSON.stringify lento em milhões de linhas | Serialização SIMD-accelerated |
| Import de CSV/SQL pesado | Streams Node.js com backpressure | Zero-copy parsing paralelo |

## Abordagem recomendada (se for seguir com Rust)

**Fase 1 — Apenas dois módulos Rust:**
1. **Export engine** — Serialização de resultsets grandes para CSV/JSON/SQL
2. **SQL parser** — `sqlparser-rs` via napi-rs para autocomplete e análise de queries

**Não migrar:**
- Drivers de conexão (`pg`, `mysql2`, `ioredis`, `@clickhouse/client`) — já são nativos
- Pool management — `kysely-factory.ts` atual funciona bem
- Schema introspection — Kysely já faz isso com type-safety
- AI/streaming — já funciona via IPC events

## Alternativas sem Rust (mais simples)

1. **MessagePort transferables** — Transferir ArrayBuffers entre main/renderer sem cópia
2. **WASM** — Compilar `sqlparser-rs` para WASM se SQL parsing virar bottleneck
3. **Worker threads** — Paralelizar processamento pesado dentro do próprio Node.js
4. **Streaming com backpressure** — Já disponível e suficiente para import/export

## Conclusão

O relatório está bem escrito e identifica corretamente os potenciais benefícios. Porém, superestima o ganho de migrar drivers (que já são nativos) e subestima a complexidade de build/testes. Se for seguir com Rust, comece apenas com export de dados e SQL parsing — dois módulos isolados que não requerem reescrever toda a camada de banco.

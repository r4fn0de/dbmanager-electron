# Plano: Otimizações na Camada de Query Engine (TypeScript)

**Objetivo**: Melhorar performance, UX e robustez da camada de banco de dados sem reescrever para Rust. Foco em caching inteligente, streaming de grandes resultados, connection pooling e query cancellation.

**Prioridade geral**: Cache > Streaming/Limites > Pooling > Cancellation

---

## Fase 1: Query Caching Inteligente (TanStack Query)

**Impacto**: Alto. Schema fica instantâneo ao trocar de aba. DDL reflete imediatamente na UI.  
**Esforço**: Médio. Refatoração de ~15 arquivos, mas sem mudanças arquiteturais.

### 1.1 Extrair QueryClient singleton

**Arquivos**: `src/app.tsx`, `src/lib/query-client.ts` (novo)

Hoje o `QueryClient` é instanciado inline no `App` component e não é exportado. Nenhum código fora do React tree pode invalidar cache.

- Criar `src/lib/query-client.ts` exportando `queryClient` singleton.
- `app.tsx` importa de lá.
- Todos os `db-actions.ts` que fazem DDL importam o singleton para `invalidateQueries`.

### 1.2 Criar `queryOptions` factories

**Arquivo**: `src/lib/query-options.ts` (novo)

Centralizar todas as `queryKey` + `queryFn` + `staleTime` em factories tipadas. Hoje `staleTime` está replicado em ~15 lugares com valores inconsistentes.

Factories necessárias:
- `schemaSummary(connectionId)` → staleTime 5min, gcTime 30min
- `databaseInfo(connectionId)` → staleTime 1min
- `tableDetails(connectionId, schema, table)` → staleTime 2min
- `tableRows(connectionId, schema, table, page, pageSize, sort, filters)` → staleTime 5min (já tem keepPreviousData)
- `schemaDefinitions(connectionId, schema, type)` → staleTime 10min (enums, functions, triggers, constraints, indexes)
- `connectionList()` → staleTime 5min, com invalidação manual após save/delete
- `localDatabases()` → staleTime 5min, com invalidação manual após create/start/stop/delete

### 1.3 Refatorar componentes para usar factories

**Arquivos**:
- `src/routes/database/-DatabasePageContent.tsx` — `schemaSummaryQuery`, `databaseInfoQuery`
- `src/features/database/components/DefinitionsBrowserPanel.tsx` — 5 queries de definições
- `src/features/database/components/TableDataEditor/TableDataEditor.tsx` — `tableRows` query
- `src/features/connection/hooks/useConnectionsList.ts` — `connections` query
- `src/features/localDb/hooks/useLocalDatabases.ts` — `localDbs` query

Cada um substitui `useQuery({ queryKey: [...], queryFn: ..., staleTime: ... })` por `useQuery(dbQueryOptions.xxx(...))`.

### 1.4 Invalidar cache após DDL

**Arquivo**: `src/features/database/hooks/db-actions.ts`

Todas as funções DDL (`createTable`, `dropTable`, `renameTable`, `addColumn`, `dropColumn`, `renameColumn`, `alterColumnType`, `setColumnNullable`, `setColumnDefault`, `createIndex`, `dropIndex`, `createSchema`) devem, após sucesso da chamada oRPC, invalidar as queries afetadas via `queryClient.invalidateQueries()`.

Mapeamento de invalidação:
- `dropTable`/`renameTable`/`createTable` → invalidate `schema-summary`, `table-details`
- `addColumn`/`dropColumn`/`renameColumn`/`alterColumnType`/`setColumnNullable`/`setColumnDefault` → invalidate `table-details` da tabela modificada
- `createIndex`/`dropIndex` → invalidate `schema-indexes`, `table-details`
- `createSchema` → invalidate `schema-summary`

Também manter as chamadas existentes a `invalidateTableCache` / `recordDdlOperation` da AI.

### 1.5 Persistir cache de schema entre sessões

**Arquivo**: `src/lib/query-client.ts`

Instalar `@tanstack/react-query-persist-client` (se não estiver) e configurar `persistQueryClient` com:
- Storage: `electron-store` via custom storage adapter (Electron localStorage tem limite de ~5–10MB; `electron-store` já é dependência do projeto e usa JSON no filesystem)
- Key: `"tarsdb-query-cache"`
- **NÃO persistir** queries `table-rows` (dados podem ser grandes)
- **NÃO persistir** queries `executeQuery` (SQL editor results)
- Persistir apenas: `schema-summary`, `db-info`, `table-details`, `schema-*`, `connections`, `local-databases`

Isso faz com que ao reabrir o app e voltar a uma conexão, o schema sidebar apareça instantaneamente.

### 1.6 Cache de EXPLAIN no SQL Editor

**Arquivo**: `src/features/database/components/SqlEditor/SqlEditor.tsx`

Para queries EXPLAIN idênticas, usar `useQuery` com `enabled: false` e `staleTime: 5min` em vez de chamar `executeQuery` diretamente. Reduz round-trip quando o usuário pressiona Ctrl+E repetidamente na mesma query.

## Fase 2: Streaming / Limitação de Resultados Grandes

**Impacto**: Alto. Evita que `SELECT * FROM huge_table` congele o app.  
**Esforço**: Baixo. Mudança localizada nos handlers.

### 2.1 Adicionar limite de segurança no handler `executeQuery`

**Arquivos**: `src/ipc/db/handlers.ts`, `src/ipc/db/types.ts`, `src/ipc/db/schemas.ts`

- Adicionar `truncated?: boolean` e `totalRowCount?: number` ao tipo `QueryResult`.
- No handler `executeQuery`, após obter resultado do driver, verificar se `rows.length > 50_000`.
- Se sim, truncar para 50k rows, setar `truncated: true`, `totalRowCount` com valor real.
- Retornar objeto enriquecido.

### 2.2 Indicar truncamento na UI do SqlEditor

**Arquivo**: `src/features/database/components/SqlEditor/SqlEditor.tsx`, `src/features/database/components/QueryResults.tsx`

- `SqlEditor` recebe o `QueryResult` e detecta `truncated === true`.
- Mostra banner/warning: "Results truncated to 50,000 rows. Add LIMIT clause to reduce result set."
- `QueryResults` (ou componente de resultados) mostra `row_count` e `totalRowCount` quando disponível.

### 2.3 Limitar também em `tableListRows`

**Arquivo**: `src/ipc/db/handlers.ts`

O `TableDataEditor` já pagina, mas `listRows` com `pageSize` muito alto (ex: 10k) ainda pode travar. Adicionar `MAX_PAGE_SIZE = 1000` no handler, clampando o input.

### 2.4 (Opcional futuro) Cursor-based streaming

**Não incluir nesta fase.** Se 50k limit não for suficiente para o caso de uso (exportação massiva), avaliar `pg-query-stream` + canal IPC paralelo. Mas isso é trabalho alto e provavelmente desnecessário para um DB client desktop.

## Fase 3: Connection Pooling Agressivo

**Impacto**: Médio. Queries paralelas (schema + table data + definitions) não mais enfileiram.  
**Esforço**: Muito baixo. Mudar 2–3 números e adicionar event listeners.

### 3.1 Aumentar pool sizes

**Arquivo**: `src/ipc/db/kysely-factory.ts`

PostgreSQL:
```ts
new PgPool({
  connectionString,
  max: 10,                    // era 2
  idleTimeoutMillis: 30_000,
  connectionTimeoutMillis: 5_000,
  keepAlive: true,
  keepAliveInitialDelayMillis: 10_000,
})
```

MySQL:
```ts
mysql.createPool({
  uri: connectionString,
  connectionLimit: 10,        // era 2
  queueLimit: 0,              // sem limite de fila
  acquireTimeout: 5_000,
  idleTimeout: 30_000,
  enableKeepAlive: true,
})
```

ClickHouse já é HTTP-based, não tem pool tradicional.
SQLite (better-sqlite3) é síncrono, não aplica.
Redis (ioredis) já tem pooling interno decente.

### 3.2 Health check logging

**Arquivo**: `src/ipc/db/kysely-factory.ts`

Adicionar listeners `pool.on("error", ...)` e `pool.on("connect", ...)` no PG pool para logar problemas de conexão. Útil para debug.

### 3.3 Warmup de conexão

**Arquivo**: `src/ipc/db/handlers.ts` (função `testConnection`)

Reutilizar o fluxo de `testConnection` existente como warmup. Quando o usuário clica "Test Connection" no form ou a aba é ativada, o `testConnection` já roda `SELECT 1` via pool, aquecendo a primeira conexão de forma natural. Não adicionar warmup automático em `resolveConnectionString` — isso pode mascarar erros de conexão se o pool não estiver fully initialized.

## Fase 4: Query Cancellation via AbortSignal

**Impacto**: Médio. Usuário pode cancelar queries longas sem matar a UI.  
**Esforço**: Alto. Requer novo canal IPC, registry de queries no main, e modificação de todos os drivers.

### 4.1 Adicionar canal de cancelamento no preload

**Arquivo**: `src/features/shell/preload.ts`

Adicionar ao `contextBridge.exposeInMainWorld("electron", { ... })`:
```ts
dbCancel: {
  cancelQuery: (requestId: string) => ipcRenderer.send("db:query:cancel", { requestId }),
}
```

### 4.2 Adicionar constantes IPC

**Arquivo**: `src/constants/index.ts`

Adicionar:
```ts
export const DB_IPC_CHANNELS = {
  QUERY_CANCEL: "db:query:cancel",
} as const;
```

### 4.3 Handler de cancelamento no main process

**Arquivo**: `src/features/shell/main.ts`

Registrar listener:
```ts
ipcMain.on(DB_IPC_CHANNELS.QUERY_CANCEL, (_event, { requestId }) => {
  cancelActiveQuery(requestId);
});
```

### 4.4 Registry de queries ativas

**Arquivo**: `src/ipc/db/active-queries.ts` (novo)

```ts
interface ActiveQuery {
  requestId: string;
  dbType: DatabaseType;
  cancel: () => void | Promise<void>;
}

const activeQueries = new Map<string, ActiveQuery>();

export function registerQuery(...)
export function cancelQuery(requestId: string): Promise<boolean>
export function unregisterQuery(requestId: string)
```

### 4.5 Modificar drivers para suportar cancelamento

**Arquivos**: `src/ipc/db/pg-runtime.ts`, `src/ipc/db/mysql-client.ts`, `src/ipc/db/clickhouse-client.ts`, `src/ipc/db/sqlite-driver.ts`

Cada driver precisa de uma variante `executeXxxQueryCancellable(requestId, ...)`:

**PostgreSQL**: usar `pg_cancel_backend(pg_backend_pid())` via conexão paralela, ou `client.query("SELECT pg_cancel_backend(...)")`.
**MySQL**: adquirir uma conexão **dedicada fora do pool** para queries canceláveis. Se o usuário cancelar, chamar `connection.destroy()` nessa conexão isolada. Isso evita poluir o pool com conexões mortas (`PROTOCOL_CONNECTION_LOSED`).
**ClickHouse**: abortar o request HTTP (usar `AbortController` no client HTTP).
**SQLite**: não aplica (queries são síncronas e locais, geralmente rápidas).

### 4.6 Modificar interface `DatabaseDriver`

**Arquivo**: `src/ipc/db/driver.ts`

Adicionar parâmetro opcional `requestId?: string` em `executeQuery`.

### 4.7 Modificar handler oRPC `executeQuery`

**Arquivo**: `src/ipc/db/handlers.ts`

Adicionar `requestId: z.string().optional()` ao schema `executeQuerySchema`. Passar `requestId` para o driver.

### 4.8 Integrar com SqlEditor AbortController

**Arquivo**: `src/features/database/components/SqlEditor/SqlEditor.tsx`

- Gerar `requestId` único por execução (`crypto.randomUUID()`).
- Passar `requestId` para `executeQuery(selectedConnection, statement, requestId)`.
- No `handleCancel`, além de `controller.abort()`, chamar `window.electron.dbCancel?.cancelQuery(requestId)`.
- O `AbortController` do renderer continua servindo para ignorar o resultado quando a promise resolve (race condition).

### 4.9 Statement timeout como fallback

**Arquivo**: `src/ipc/db/pg-runtime.ts`

Para PostgreSQL, adicionar `SET statement_timeout = 30000` (30s) antes de cada query. Se o cancelamento não funcionar, o banco mesmo aborta. MySQL não tem statement_timeout nativo equivalente.

## Ordem de Implementação

1. **Fase 3** (pooling) → menor esforço, pode ser feito em 10 minutos
2. **Fase 2** (limites) → também rápido, protege contra crashes imediatamente
3. **Fase 1** (cache) → maior esforço, mas maior impacto na UX
4. **Fase 4** (cancellation) → deixar por último, é o mais complexo

## Riscos e Mitigações

| Risco | Mitigação |
|---|---|
| QueryClient singleton quebra HMR no dev | Exportar de `src/lib/query-client.ts` e reutilizar; React Query tolera isso bem |
| Persistência de cache com schema antigo após DDL | Invalidação explícita em todos os db-actions DDL |
| Pool size 10 consome muitas conexões no servidor | Desktop app com 1–2 usuários simultâneos; 10 conexões é conservador para um client |
| Cancelamento de query MySQL destrói conexão do pool | Usar conexão dedicada fora do pool para queries canceláveis; destruir apenas a conexão isolada |
| `executeQuerySchema` alterado quebra tipagem | oRPC regenera types automaticamente; verificar `tsc --noEmit` |

## Notas

- Não adicionar novas dependências nativas (Rust, etc.). Apenas `@tanstack/react-query-persist-client` se ainda não estiver.
- Manter o padrão oRPC para tudo. A única exceção de IPC raw é o canal de cancelamento, que segue o mesmo padrão dos canais de AI streaming já existentes.
- Não modificar `src/components/ui/` (regra do projeto).
